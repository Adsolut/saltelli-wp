# PHASE 1 — JSX Inventory (15 file)

## Riassunto in una riga

**I 15 JSX NON sono nuovi design da implementare da zero — sono le _design source_ delle pagine
già costruite (Sessione 1 + Sessione 2 + Logo v1.1, ovvero il tema dal v0.x al v1.3.x).** Il tema
WordPress contiene già `.sl-blog2__*`, `.sl-404__*`, `.sl-hero__*`, `.sl-area`, `.sl-attorney__*`,
`.sl-tier1__*`, `.sl-tipoarea__*`, `.sl-glossario__*`, `.sl-foot-*` — un commento in testa a quasi
ogni JSX cita esplicitamente il file `.claude/knowledge/design/sessione-N/` di provenienza. Quindi
il bundle è un **re-handoff** dei sorgenti di design → l'audit serve a misurare il **drift** fra
questi prototipi e ciò che è già a frontend, non a pianificare un build greenfield.

## Note trasversali (valgono per ≥10 JSX)

1. **Header/footer condivisi** — Sessione 2 (#7–#15) importa `<S2Header>` + `<S2Footer>` da `chrome/index.jsx`/`footer/index.jsx` via global window. Il footer "vero" è `footer/index.jsx` (4 fasce dense); il footer dentro `chrome/index.jsx` e quello inline in `home/index.jsx` sono la versione **3-col vecchia** — vanno ignorati a favore di `footer/index.jsx`.

2. **Navigazione hardcoded con URL legacy** — `chrome` (e `footer/index.jsx`) usano slug pre-Wave-5: `/avvocati/`, `/competenze/`, `/casi/`, `/editoriale/`, `/blog/`, `/costi/`, `/prima-consulenza/`, `/come-lavoriamo/`, `/chi-siamo/risultati/`. Tutti hanno già un redirect 301 in `inc/redirects.php`, ma il **markup deve usare gli slug correnti** (`/chi-siamo/team/`, `/aree-di-pratica/`, `/chi-siamo/casi-rappresentativi/`, `/risorse/blog/`, `/costi-e-consulenze/...`). In WP la nav primaria è `wp_nav_menu('primary')` (admin-managed): le voci JSX servono come _checklist del menu atteso_, non da copiare letteralmente.

3. **Dati hardcoded = placeholder di design** — gli array `areas`/`lawyers`/`cases`/`terms`/`posts` nel JSX sono placeholder. In WP il contenuto dinamico viene da CPT/taxonomy (`competenza` ×19, `avvocato` ×4, `saltelli_caso`, `category` blog) — già implementato. Eccezione: il glossario è hardcoded in `inc/wave3-glossario.php` come array PHP (stesso sorgente del JSX), per scelta — non un CPT.

4. **Niente file immagine** — confermato dal README e dal grep: zero riferimenti `.jpg/.png/.webp/.avif/.svg` nei JSX. Le immagini sono placeholder CSS (gradient diagonale `linear-gradient(135deg, #c8c5be 0%, #6e6c66 100%)` + filtro `grayscale(1) contrast(1.05)` che diventa `grayscale(0)` su hover, oppure striped `repeating-linear-gradient`). Gli screenshot in `_reference/screenshots/` sono concept di hero **logo-stack** (4 varianti), non asset da uploadare. → Le immagini reali sono backlog Wave 5.1, fuori scope.

5. **Pattern interattivi (React state → PHP/JS in WP)** — già tradotti nel tema: header scroll→solid (`data-scrolled` inline script in `header.php`), hero word-reveal (`data-split-reveal` + `saltelli_split_h1_words()` + GSAP/`main.js`), filtro pillole aree (`data-filter` + `main.js`), area-hover→preview pane (`data-area-preview` + `main.js`), accordion (`details/summary` o `data-open`), tab categorie blog (link `?cat=` **server-side** — corretto per SEO, non React state), search glossario + A-Z jump (`main.js`/`wave3-glossario.php`), form contatti + newsletter Brevo (POST). Il JSX usa lo stato React solo perché è un prototipo: nessuna conversione client-side va introdotta lato WP.

6. **Token usati nei JSX ≠ `tokens-design-bundle.css`** — le `h1`/hero usano `clamp()` inline con i valori **GRANDI** del sistema corrente (`clamp(80px,9vw,132px)`, `clamp(64px,8vw,132px)`, `clamp(72px,9vw,140px)`, `clamp(56px,6vw,88px)`…), non i valori piccoli del bundle (`clamp(48px,8vw,120px)`, `clamp(36px,5vw,64px)`). Il bundle CSS è quindi un artefatto **preliminare/obsoleto**; i JSX riflettono un modello mentale più vicino a `tokens.css` corrente (Wave 5 STEP 2). Dettaglio in `03-tokens-reconciliation.md`.

7. **`design-system/index.jsx` non è una pagina** — è la showcase di reference dei token. Usa i valori vecchi. Non mappa a nessun template WP (non esiste `/design-system/`). Informativo. Eventuale pagina interna `/design-system/` = backlog post-cut, bassissima priorità.
