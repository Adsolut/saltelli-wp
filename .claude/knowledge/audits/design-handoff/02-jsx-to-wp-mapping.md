# PHASE 2 — JSX → WP Theme Mapping

## Legenda classificazione

**Visual gap** (JSX vs ciò che il tema renderizza oggi):
- ✅ **Già presente** — il template WP ha la struttura matching, serve solo verifica + eventuale drift cleanup CSS
- ⚠️ **Parziale** — la struttura c'è ma manca una sezione/elemento specifico del JSX
- ❌ **Mancante** — il JSX introduce una feature non presente nel tema
- 🔄 **Refactor** — il JSX cambia struttura/ordine → valuta SCF impact

**SCF Data Contract impact** (hard invariant — vedi prompt §"⚠️ HARD INVARIANT"):
- 🟢 **COMPLIANT** — legge da field SCF esistenti / non tocca SCF → zero schema change
- 🟡 **ADDITIVE** — servono field SCF nuovi su group esistente o group nuovo → safe
- 🔴 **RE-INTERPRETATION** — il JSX implica refactor schema/location/group/Gutenberg/admin-path → BLOCCA, rinegoziare con Design

**Nessuna RE-INTERPRETATION tocca: schema SCF, location rules, group structure, Gutenberg pattern sui 13 Pages SCF-only, admin path, CPT/taxonomy registration.** L'unica RE-INTERPRETATION è una decisione di prodotto (mappa sì/no), non un refactor strutturale. Il "mapping ambiguity" chi-siamo→lo-studio è una decisione di routing, anch'essa non strutturale.
