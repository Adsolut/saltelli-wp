## Decisioni autonome (da reportare)

1. **`inc/cpt-recovery.php` invece di `inc/cpt.php`**
   Il prompt indicava di "modificare `inc/cpt.php`" ma quel file non esiste. La convention reale del theme è `cpt-{name}.php` (vedi `cpt-avvocato.php`, `cpt-competenza.php`). Creato file separato + include in `functions.php` riga 21.

2. **`inc/acf-json/` riusato (non duplicato in root theme)**
   Il prompt indicava di creare `wp-content/themes/saltelli/acf-json/` ma esiste già `wp-content/themes/saltelli/inc/acf-json/` con 3 field group pre-esistenti (`group_avvocato.json`, `group_competenza.json`, `group_settings.json`) e già hookato in `acf-fields.php`. Riusato senza duplicazione.

3. **Skip `unset($paths[0])` su load_json filter**
   `acf-fields.php` lascia il default path nei load paths (commento esplicito "opzionale: lo lasciamo per retro-compat"). Mantenuto comportamento esistente.

4. **Permission fix droplet pre-install**
   Su droplet, prima dell'install ACF, due fix necessari (eseguiti via sudo):
   - `chown -R www-data:www-data /var/www/.wp-cli` (creato per WP-CLI cache)
   - `chown -R www-data:www-data wp-content/plugins wp-content/upgrade` + `chmod g+w` (era owner deploy:www-data senza group write — WP-CLI non poteva installare plugin)
   Da considerare per provisioning futuro nel runbook G.
